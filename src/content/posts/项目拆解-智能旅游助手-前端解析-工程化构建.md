7.1 Vite 配置深度解析

  

```js

export default defineConfig({

  plugins: [

    vue(),

    Components({ resolvers: [VantResolver()] }),

  ],

  server: {

    proxy: {

      '/api': {

        target: 'http://localhost:3300',

        changeOrigin: true,

      },

    },

  },

})

```

  

#### 7.1.1 `@vitejs/plugin-vue` 做了什么？

  

1. **SFC 编译**：将 `.vue` 文件拆分为 `<template>` → `render` 函数、`<script>` → ES module、`<style>` → CSS

2. **HMR**：组件级热更新，修改 `.vue` 文件后仅替换该组件，不刷新页面

3. **`<script setup>` 编译**：将 `ref`、`computed` 等编译时语法糖转换为标准 JS

4. **`defineProps`/`defineEmits` 编译时宏**：无需 import 即可使用的编译器宏

  

#### 7.1.2 `unplugin-vue-components` 按需导入原理

  

```

编译时转换示例：

  

输入模板：

  <van-button type="primary">点击</van-button>

  <van-icon name="home-o" />

  

输出（自动注入 imports）：

  import { Button, Icon } from 'vant'

  // 模板中对应标签被替换为组件引用

```

  

**VantResolver 的工作：**

```js

// 伪代码

{

  resolve(name) {

    if (name.startsWith('Van')) {

      // VanButton → import { Button } from 'vant'

      return { from: 'vant', name: name.slice(3) }

    }

  }

}

```

  

#### 7.1.3 开发代理 —— 解决跨域

  

```

浏览器 → localhost:5173/api/plan

              │

              ▼ (Vite Dev Server 拦截)

        匹配到 /api 前缀

              │

              ▼ (转发)

         localhost:3300/api/plan

              │

              ▼ (Express 处理)

         返回 JSON

```

  

**`changeOrigin: true` 的作用：**

- 修改请求头中的 `Host` 为目标 URL 的 host

- 避免后端根据 `Host` 头做校验时拒绝请求

### 9.2 LangChain + DeepSeek 的适配模式

  

```js

// 核心：通过 configuration.baseURL 桥接到 DeepSeek

this.llm = new ChatOpenAI({

  openAIApiKey: apiKey,

  configuration: { baseURL: 'https://api.deepseek.com/v1' },  // ← 关键！

  modelName: 'deepseek-chat',

  temperature: 0.7,

})

```

  

**这是面试亮点 —— "适配器模式"的实践：**

- DeepSeek API 兼容 OpenAI SDK 格式

- LangChain 的 `ChatOpenAI` 可以无缝切换到任何 OpenAI 兼容的 API

- 换模型只需修改 `baseURL` 和 `modelName`，代码零改动

  

### 9.3 两层通信模式对比

  

| 特性 | `/api/plan` | `/api/chat` |

|------|-------------|-------------|

| **协议** | HTTP JSON | HTTP SSE (Server-Sent Events) |

| **前端** | `axios.post()` | `fetch()` + `ReadableStream` |

| **后端** | `res.json()` | `res.write()` 逐块推送 |

| **LangChain** | `llm.invoke()` | `llm.stream()` |

| **用户体验** | 全量返回后一次性展示 | 逐字打字机实时呈现 |

| **可取消** | AbortController | reader.cancel() + AbortController |

| **适用场景** | 完整规划生成 | 对话交互 |

  

### 9.4 Markdown 正则解析 —— 三层降级策略

  

这是后端最有技术含量的部分：

  

```js

_parseSlots(block) {

  // 策略1：表格格式

  const tableSlots = block.matchAll(/\|\s*(上午|中午|下午|晚上).../g)

  if (tableSlots.length > 0) return tableSlots.map(...)

  

  // 策略2：列表格式（带子项）

  const slotRe = /[-*]\s+\*{1,2}(上午|中午|...)[：:][^*]*?\*{1,2}\s*([\s\S]*?).../g

  // ...

  

  // 策略3：简单行格式

  const fallRe = /[-*]\s+\*{1,2}(上午|中午|...)[：:][^*]*?\*{1,2}\s*([^\n]+)/g

  // ...

}

```

  

**设计意义：** AI 输出格式不可控，三层降级保证了即使 AI 返回格式不规范，也能最大程度提取结构化数据。
