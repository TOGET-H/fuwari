## 二、技术选型深度分析

  

### 2.1 为什么选 Vue 3 而不是 React？

  

| 维度 | Vue 3 | React 18/19 |

|------|-------|-------------|

| **学习曲线** | 低，模板语法接近 HTML，`<script setup>` 简洁 | 中，JSX、Hooks 规则、闭包陷阱 |

| **响应式系统** | `Proxy` 自动追踪，`ref`/`reactive` 精细粒度 | 手动 `useState`/`useMemo`，需理解不可变数据 |

| **移动端生态** | Vant 4 成熟稳定 | 无官方移动端方案（Ant Design Mobile 等社区方案） |

| **SSR** | Nuxt 3 完善 | Next.js 14+ 成熟 |

| **TypeScript** | 良好支持 | 原生 JSX 类型推导更好 |

| **包体积** | ~16KB gzip（运行时） | ~42KB gzip（react + react-dom） |

  

**本项目选择 Vue 3 的核心原因：**

- 目标是**移动端 SPA**，Vant 组件库与 Vue 生态天然契合

- Composition API 的 `<script setup>` 语法让单文件组件代码更紧凑

- 不需要 SSR，纯客户端渲染满足需求

  

### 2.2 为什么用 Vant 4？

  

**Vant 是 Vue 移动端第一选择**，原因：

  

1. **轻量**：组件按需加载，配合 `unplugin-vue-components` 自动 tree-shaking

2. **移动端原生体验**：内置手势、触摸反馈、安全区适配（`safe-area-inset-bottom`）

3. **设计一致性**：统一的 Design Token 体系，本项目在此基础上做了渐变主题定制

  

**面试延伸 —— 按需导入原理：**

### 2.4 为什么用 Axios + Fetch 混用？

### 3.2 核心面试点

#### 3.2.1 `createWebHistory` vs `createWebHashHistory`

  

| 对比维度 | WebHistory | WebHashHistory |

|----------|------------|----------------|

| URL 形式 | `/chat` | `/#/chat` |

| 原理 | H5 `history.pushState` API | `window.location.hash` 变更 |

| SEO | 友好（URL 干净） | 不友好（`#` 被忽略） |

| 服务器配置 | **必须**配置 fallback 到 `index.html` | 不需要（hash 不会发到服务器） |

| 兼容性 | IE10+ | 所有浏览器 |

  

**本项目选 `createWebHistory` 的理由：**

- URL 美观，无 `#` 号

- 部署时配置 Nginx `try_files $uri /index.html` 即可解决刷新 404

  

**面试追问：为什么 History 模式刷新会 404，Hash 不会？**

  

```

用户访问 /chat → 刷新浏览器

  │

  ├─ History 模式：浏览器向服务器请求 /chat

  │     → 服务器没有 /chat 这个物理文件 → 404

  │     → 解决方法：Nginx try_files $uri /index.html（所有路径 fallback 到 SPA 入口）

  │

  └─ Hash 模式：浏览器请求 / （# 及之后内容不发到服务器）

        → 服务器返回 index.html → Vue Router 解析 hash 渲染对应页面 → 正常

```

#### 3.2.2 路由懒加载原理

  

```js

// 静态导入 —— 打包到一个 chunk，首屏全部加载

import Home from '../views/Home.vue'

  

// 动态导入 —— 每个页面独立 chunk，按需加载

const Home = () => import('../views/Home.vue')

```

  

**Vite 处理 `import()` 的流程：**

  

```

1. 编译时：Vite 识别到 import() 调用

2. 代码分割：Rollup 将每个 import() 的模块拆成独立 chunk

3. 运行时：首次访问路由时，动态创建 <script> 标签加载对应 chunk

4. 缓存：加载后的 chunk 被浏览器缓存，二次访问无需重新下载

```

#### 3.2.3 导航守卫 vs 组件内守卫

  

**本项目未使用全局守卫**，但这是面试高频考点。完整守卫执行顺序：

  

```

1. 导航触发

2. 在失活组件里调用 beforeRouteLeave        ← 组件内守卫

3. 调用全局 beforeEach                      ← 全局前置守卫

4. 在重用组件里调用 beforeRouteUpdate        ← 组件内守卫（动态路由参数变化）

5. 在路由配置里调用 beforeEnter              ← 路由独享守卫

6. 解析异步路由组件

7. 在被激活组件里调用 beforeRouteEnter       ← 组件内守卫

8. 调用全局 beforeResolve                   ← 全局解析守卫

9. 导航被确认

10. 调用全局 afterEach                       ← 全局后置钩子

11. 触发 DOM 更新

12. 调用 beforeRouteEnter 中传给 next 的回调

```

#### 3.2.4 `router.push` 传参方式对比

  

本项目使用了两种传参方式：

| 方式 | URL 体现 | 刷新保留 | 适用场景 |

|------|---------|---------|----------|

| `query` | `/details?destination=三亚&budget=5000` | ✅ 保留 | 搜索参数、筛选条件 |

**面试追问：`params` 和 `query` 的本质区别？**

- `params` 是路由的一部分，定义在 `path: '/user/:id'` 中，必须匹配才能进入路由

- `query` 是 URL 的附加参数，不影响路由匹配，可选

**面试重点：为什么要在 `onBeforeUnmount` 清理？**

  

1. **内存泄漏**：`setInterval` 不清理会在组件销毁后继续执行，引用已销毁的组件状态导致内存泄漏

2. **SSE 连接**：不关闭会导致服务器持续推送数据到已不存在的页面

3. **Vue 3 特点**：`<script setup>` 中没有 `this`，所有变量都是闭包，定时器回调持有闭包引用，阻止 GC

### 6.2 SSE 流式请求 —— 本项目的技术亮点

  

这是**全手写的 SSE 客户端实现**，不依赖任何库。

  

#### 6.2.1 SSE 协议格式

  

```

data: {"content":"你好"}

  

event: end

data: {"done":true}

  

event: error

data: {"error":"服务端错误"}

```

  

- 每个事件块以 `\n\n` 分隔

- `event:` 行定义事件类型

- `data:` 行承载 JSON 数据

  

#### 6.2.2 客户端实现逐步解析

  

```js

export function stream(url, data, onChunk, onComplete, onError, signal) {

  let canceled = false

  let reader = null

  

  // ① 返回取消函数（闭包模式）

  const cancel = () => {

    canceled = true

    if (reader) {

      try { reader.cancel() } catch (_) {}

    }

  }

  

  // ② IIFE 异步执行，不阻塞调用方

  ;(async () => {

    try {

      const response = await fetch(`/api${url}`, {

        method: 'POST',

        headers: { 'Content-Type': 'application/json' },

        body: JSON.stringify(data),

        signal,  // ← 传入 AbortSignal，支持外部取消

      })

  

      // ③ 获取流读取器

      reader = response.body.getReader()

      const decoder = new TextDecoder()  // UTF-8 解码器

      let buffer = ''

  

      while (!canceled) {

        const { done, value } = await reader.read()

        if (done) {

          // 处理缓冲区残留

          if (buffer.trim()) {

            const data = parseSSELine(buffer.trim())

            if (data) onChunk(data)

          }

          onComplete()

          return

        }

  

        // ④ 增量解码 + 缓冲区拼接

        buffer += decoder.decode(value, { stream: true })

  

        // ⑤ 按 \n\n 分割出完整事件块

        const parts = buffer.split('\n\n')

        buffer = parts.pop() || ''  // 最后一个可能不完整，保留在 buffer

  

        // ⑥ 解析每个事件块

        for (const block of parts) {

          // 逐行解析 event: 和 data:

          let eventType = ''

          let eventData = null

          for (const line of block.split('\n')) {

            if (line.startsWith('event: ')) {

              eventType = line.slice(7).trim()

            } else if (line.startsWith('data: ')) {

              try { eventData = JSON.parse(line.slice(6)) } catch (_) {}

            }

          }

  

          if (eventType === 'error') throw new Error(eventData.error)

          if (eventType === 'end' || eventData.done) { onComplete(); return }

          onChunk(eventData)

        }

      }

    } catch (e) {

      if (e.name === 'AbortError' || canceled) {

        onComplete()

        return

      }

      onError(e)

    }

  })()

  

  return cancel

}

```

  

#### 6.2.3 打字机效果实现

  

```js

// Chat.vue 中的打字机系统

  

// 字符队列 —— 模块级变量，非响应式（高频操作避免响应式开销）

let typeQueue = []

let typeTimer = null

  

// 启动打字机：每 20ms 从队列取一个字符追加到消息

function startTypewriter(aiMsg) {

  typeQueue = []

  if (typeTimer) clearInterval(typeTimer)

  

  typeTimer = setInterval(() => {

    if (typeQueue.length === 0) return

    aiMsg.content += typeQueue.shift()  // ← 直接操作响应式对象的属性

    scrollToBottom()

  }, 20)  // 20ms ≈ 50 字/秒，模拟真实打字速度

}

  

// SSE 回调 → 字符入队

function feedChars(text) {

  for (const ch of text) {

    typeQueue.push(ch)

  }

}

  

// 停止打字机：flush 剩余队列

async function stopTypewriter(aiMsg) {

  if (typeTimer) {

    clearInterval(typeTimer)

    typeTimer = null

  }

  if (typeQueue.length > 0) {

    aiMsg.content += typeQueue.join('')  // 一次性输出剩余字符

    typeQueue = []

  }

  await nextTick()

  scrollToBottom()

}

```

  

**设计精妙之处：**

  

1. **解耦**：SSE 数据到达 (`feedChars`) 和 UI 渲染 (`setInterval`) 是两个独立过程

2. **背压处理**：如果 AI 输出快于打字机消费，字符在队列中排队，不会丢失

3. **非响应式队列**：`typeQueue` 不用 `ref`，避免高频响应式追踪的性能开销

4. **优雅终止**：`stopTypewriter` flush 剩余队列，确保不丢字
