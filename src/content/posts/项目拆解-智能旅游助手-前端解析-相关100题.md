### Vue 基础（1-20）

  

**Q1: Vue 3 的响应式原理是什么？和 Vue 2 有什么区别？**

  

> Vue 3 使用 `Proxy` 替代 `Object.defineProperty`。

> - `Proxy` 可以拦截整个对象的所有操作（get/set/deleteProperty/has 等），不需要递归遍历

> - Vue 2 的 `defineProperty` 无法检测属性添加/删除和数组索引修改

> - Vue 3 的 `ref` 内部也是 `reactive` 的封装：`{ value: xxx }` → `reactive({ value: xxx })`

> - 本项目中 `formData`（reactive 对象）和 `active`（ref 数字）体现了两种 API 的使用

  

**Q2: `ref` 和 `reactive` 的区别和使用场景？**

  

> - `ref`：包装基本类型，通过 `.value` 访问/修改；模板中自动解包；可整体替换

> - `reactive`：包装对象/数组，直接访问属性；不能整体替换；解构会丢失响应性（需 `toRefs`）

> - 本项目实践：基本类型 + 数组用 `ref`（`messages`、`inputText`），固定结构对象用 `reactive`（`formData`）

  

**Q3: `computed` 和 `watch` 的区别？**

  

> - `computed`：声明式派生状态，有缓存，依赖不变不重新计算；返回值是只读 ref

> - `watch`：命令式副作用监听，无缓存，可执行异步操作；可以监听多个源

> - 本项目 `Details.vue` 用 `computed` 从 `tripResult` 派生 `overview`、`schedule` 等

> - `App.vue` 用 `watch` 监听 `route.path` 同步 tab 状态（副作用）

  

**Q4: Vue 3 的生命周期有哪些？**

  

> 创建 → 挂载 → 更新 → 销毁

> - `setup()` — 组件初始化（Composition API 入口）

> - `onBeforeMount` — DOM 挂载前

> - `onMounted` — DOM 挂载后（本项目 Chat.vue 在此添加欢迎消息）

> - `onBeforeUpdate` — 数据变化，DOM 更新前

> - `onUpdated` — DOM 更新后

> - `onBeforeUnmount` — 组件销毁前（本项目在此清理定时器和 SSE 连接）

> - `onUnmounted` — 组件销毁后

> - `onActivated/onDeactivated` — `<keep-alive>` 专用

  

**Q5: `nextTick` 的作用和原理？**

  

> 作用：在下一次 DOM 更新循环结束后执行回调。

> 原理：Vue 内部维护一个异步更新队列，多次数据变更会被合并为一次 DOM 更新。`nextTick` 将回调推入微任务队列（Promise.then），在 DOM 更新后执行。

> 本项目 `scrollToBottom()` 中使用 `nextTick` 确保获取到最新 DOM 高度。

  

**Q6: `v-if` 和 `v-show` 的区别？**

  

> - `v-if`：条件为 false 时**不渲染**元素（DOM 不存在），切换开销大，适合不频繁切换

> - `v-show`：始终渲染，条件为 false 时 `display: none`，初始开销大，适合频繁切换

> - 本项目 `Details.vue` 用 `v-if="typing"` 控制打字机光标显示，用 `v-if="schedule.length"` 控制列表渲染

  

**Q7: `v-for` 中 `key` 的作用？为什么不能用 index？**

  

> `key` 是 Vue 虚拟 DOM diff 时的唯一标识，用于判断节点是否可复用。

> 用 index 的问题：当列表增删/排序时，同一 index 可能对应不同数据，导致错误复用（输入框内容残留等问题）。

> 本项目 Chat.vue 的消息列表使用 `:key="idx"`，因为消息列表只有追加操作无插入/删除/排序，用 index 是可接受的。

  

**Q8: Vue 组件间通信有哪些方式？**

  

> 1. Props + Emits（父子）

> 2. `v-model`（父子双向绑定）—— 本项目 `van-tabbar v-model="active"`

> 3. `provide/inject`（祖先→后代依赖注入）

> 4. 路由参数 query/params —— 本项目 Home → Details

> 5. Pinia/Vuex（全局状态）

> 6. EventBus（`mitt` 库，Vue 3 已移除 $on/$off）

> 7. `$refs` / `defineExpose`（父访问子）

  

**Q9: `watchEffect` 和 `watch` 的区别？**

  

> - `watchEffect`：自动追踪回调中使用的所有响应式依赖，立即执行一次

> - `watch`：需要显式指定监听源，可配置 `immediate`、`deep` 等选项

> - 本项目的 `watch(route.path, ..., { immediate: true })` 可以改为 `watchEffect`，但显式监听更清晰

  

**Q10: Vue 3 的 `<script setup>` 相比 Options API 有什么优势？**

  

> 1. 代码更简洁，减少样板代码（不需要 `data()`/`methods`/`computed` 等）

> 2. 更好的类型推导（TypeScript 友好）

> 3. 逻辑复用更方便（Composables 函数）

> 4. 编译时优化（静态分析更准确）

> 5. 更好的 tree-shaking

> 本项目全量使用 `<script setup>`

  

**Q11: 什么是 Composition API 的 "Composable" 函数？**

  

> 将可复用的有状态逻辑提取成独立函数，命名以 `use` 开头。例如本项目可以抽取：

```js

// composables/useTypewriter.js

export function useTypewriter() {

  let queue = [], timer = null

  function start(aiMsg) { /* ... */ }

  function feed(text) { /* ... */ }

  function stop(aiMsg) { /* ... */ }

  return { start, feed, stop }

}

// Chat.vue 和 Details.vue 都在用相似的打字机逻辑，应抽取为 composable

```

  

**Q12: `toRefs` 和 `toRef` 的区别？**

  

> - `toRefs(obj)`：将 reactive 对象的每个属性转为独立的 ref，保持响应性连接

> - `toRef(obj, 'key')`：将单个属性转为 ref

> - 使用场景：解构 reactive 对象时保持响应性

  

**Q13: Vue 中如何实现组件缓存？**

  

> 使用 `<keep-alive>` 包裹 `<router-view>`：

> ```html

> <router-view v-slot="{ Component }">

>   <keep-alive>

>     <component :is="Component" />

>   </keep-alive>

> </router-view>

> ```

> 配合 `onActivated`/`onDeactivated` 生命周期。本项目的 Chat 页面如果被缓存，离开再返回时不会丢失对话历史。

  

**Q14: `shallowRef` 和 `ref` 的区别？**

  

> - `ref`：深层响应式，对象内部所有属性变更都会触发更新

> - `shallowRef`：只有 `.value` 的替换会触发更新，内部属性变化不触发

> - 使用场景：大数据量列表、第三方对象实例，不需要深层响应式追踪

  

**Q15: Vue 的异步组件是什么？**

  

> 使用 `defineAsyncComponent` 定义的组件，加载时才请求 JS：

> ```js

> const AsyncComp = defineAsyncComponent(() => import('./Heavy.vue'))

> ```

> 与本项目的路由懒加载类似，但用于非路由组件。支持 loading/error 状态。

  

**Q16: `v-once` 和 `v-memo` 的作用？**

  

> - `v-once`：元素/组件只渲染一次，后续数据变化不更新（静态内容优化）

> - `v-memo`：接收依赖数组，只有依赖变化时才重新渲染（Vue 3.2+）

  

**Q17: Vue 的 `Transition` 和 `TransitionGroup` 是什么？**

  

> 内置组件，用于给元素/组件的进入/离开添加动画：

> - `Transition`：单个元素/组件的过渡

> - `TransitionGroup`：列表的过渡（需要 key）

> 本项目打字机光标的闪烁是纯 CSS `@keyframes`，没有使用 Transition 组件。

  

**Q18: Vue 的 `Teleport` 组件是什么？**

  

> 将组件的内容渲染到 DOM 树的指定位置（如 `body` 末尾），常用于 Modal/Dialog：

> ```html

> <Teleport to="body">

>   <div class="modal">...</div>

> </Teleport>

> ```

> 本项目未使用。

  

**Q19: `Suspense` 组件的作用？**

  

> 用于协调异步组件/async setup 的加载状态，显示 fallback 内容直到异步依赖就绪。

> Vue 3 实验性功能，本项目未使用。

  

**Q20: 如何优化大型 Vue 应用的性能？**

  

> 1. 路由懒加载（本项目已使用）

> 2. 组件按需导入（本项目 Vant 已使用）

> 3. `v-once` / `v-memo` 静态内容优化

> 4. `<keep-alive>` 缓存路由组件

> 5. 虚拟滚动（大量列表）

> 6. `shallowRef` 减少深层响应式开销

> 7. 合理拆分 Computed（粒度适中）

> 8. 避免在模板中使用复杂表达式

  

### Vue Router（21-35）

  

**Q21: Vue Router 的两种 History 模式有什么区别？底层原理是什么？**

  

> 见 3.2.1 节详细对比。本项目使用 `createWebHistory`。

  

**Q22: 路由懒加载是如何实现的？打包后是怎样的？**

  

> 见 3.2.2 节详细说明。本项目 4 个路由全部使用 `() => import(...)` 懒加载。

  

**Q23: Vue Router 的导航守卫有哪些？执行顺序是什么？**

  

> 见 3.2.3 节完整流程。

  

**Q24: `$route` 和 `$router` 的区别？在 Composition API 中如何获取？**

  

> - `$route`（`useRoute()`）：当前路由信息对象（只读），包含 path/params/query/meta

> - `$router`（`useRouter()`）：路由实例，用于编程式导航 push/replace/go 等

> - 本项目 `App.vue` 同时使用了二者：`useRoute()` 监听变化，`useRouter()` 执行跳转

  

**Q25: `router.push`、`router.replace`、`router.go` 的区别？**

  

> - `push`：添加新历史记录，可返回

> - `replace`：替换当前历史记录，不可返回（本项目未使用）

> - `go(n)`：在历史记录中前进/后退 n 步（等同于 `window.history.go`）

  

**Q26: 动态路由是什么？如何响应参数变化？**

  

> 路径中使用 `:paramName`，如 `/user/:id`。参数变化时，同一组件被复用，需通过 `watch` 或 `beforeRouteUpdate` 响应：

> ```js

> watch(() => route.params.id, (newId) => { /* 重新加载数据 */ })

> ```

  

**Q27: 路由元信息 `meta` 的使用场景？**

  

> 在路由配置中附加自定义数据：

> ```js

> { path: '/profile', meta: { requiresAuth: true } }

> // 在 beforeEach 中检查 meta，实现权限控制

> ```

> 本项目未使用 meta，但可以用于标记需要登录的页面。

  

**Q28: 如何实现路由切换动画？**

  

> ```html

> <router-view v-slot="{ Component, route }">

>   <transition :name="route.meta.transition || 'fade'">

>     <component :is="Component" />

>   </transition>

> </router-view>

> ```

  

**Q29: 路由的 `props` 解耦是什么？**

  

> 在路由配置中设置 `props: true`，将 `params` 作为组件的 props 传入，解耦组件和 `$route`：

> ```js

> { path: '/user/:id', component: User, props: true }

> // User.vue 中 defineProps(['id']) —— 不再依赖 useRoute()

> ```

  

**Q30: 如何在路由切换前阻止导航？**

  

> 使用 `beforeRouteLeave` 或 `beforeEach` 守卫，调用 `next(false)` 或返回 `false` 阻止。

  

**Q31: Vue Router 的滚动行为如何配置？**

  

> ```js

> createRouter({

>   scrollBehavior(to, from, savedPosition) {

>     if (savedPosition) return savedPosition  // 浏览器前进/后退

>     return { top: 0 }  // 新导航滚动到顶部

>   }

> })

> ```

  

**Q32: 如何实现标签页与路由的同步？**

  

> 本项目 `App.vue` 的做法：

> ```js

> // 路由 → Tab：watch route.path 同步 active 索引

> // Tab → 路由：onTabChange 触发 router.push

> ```

> 这是移动端 App 类布局的经典模式。

  

**Q33: `router.resolve` 的作用？**

  

> 解析目标位置，返回规范化的路由信息（包括 fullPath、matched 等），不触发导航。

> 用于需要提前知道目标路由信息的场景。

  

**Q34: 路由命名视图（Named Views）是什么？**

  

> 同一路由渲染多个视图（如侧边栏 + 主内容）：

> ```html

> <router-view name="sidebar" />

> <router-view />  <!-- default -->

> ```

> 本项目只有一个 `<router-view>`，未使用命名视图。

  

**Q35: 如何实现路由权限控制？**

  

> ```js

> router.beforeEach((to, from) => {

>   if (to.meta.requiresAuth && !store.isLoggedIn) {

>     return { path: '/login', query: { redirect: to.fullPath } }

>   }

> })

> ```

> 本项目无需权限控制（无登录体系），属于简化场景。

  

### Vite & 工程化（36-50）

  

**Q36: Vite 为什么开发冷启动快？**

  

> 1. **ESM 原生支持**：开发环境不打包，浏览器直接请求模块，按需编译

> 2. **esbuild 预构建**：依赖预构建用 Go 写的 esbuild，比 JS 打包快 10-100 倍

> 3. **按需编译**：访问哪个页面编译哪个页面，不是全量编译

> 4. **缓存策略**：已编译模块缓存到 `node_modules/.vite/`

  

**Q37: Vite 的热更新（HMR）原理？**

  

> 1. Vite Server 通过 WebSocket 与浏览器建立连接

> 2. 文件变更时，Vite 找到受影响模块的边界

> 3. 只推送变更模块的新代码（不是整个页面）

> 4. 浏览器接收后热替换对应模块，保持应用状态

> 5. Vue 组件的 HMR 由 `@vitejs/plugin-vue` 处理，粒度到组件级

  

**Q38: Vite 的预构建（Pre-bundling）做了什么？**

  

> 1. 将 CommonJS/UMD 模块转为 ESM

> 2. 将多文件依赖合并为一个文件（减少 HTTP 请求）

> 3. 例如 `lodash` 有 600+ 个模块文件，预构建后合并为一个

  

**Q39: `unplugin-vue-components` 的工作原理？**

  

> 见 7.1.2 节。编译时 AST 分析模板 → 匹配组件名 → 自动注入 import。

  

**Q40: Vite plugin 和 Webpack loader/plugin 的区别？**

  

> - Vite plugin：基于 Rollup 插件接口 + Vite 特有钩子（`configureServer`、`transformIndexHtml` 等）

> - Webpack loader：文件转换（单一职责，链式调用）

> - Webpack plugin：构建流程介入（更广泛的能力，通过 Tapable 钩子系统）

> - Vite 开发阶段用 esbuild，生产阶段用 Rollup

  

**Q41: Tree Shaking 的原理？**

  

> 基于 ES Module 的静态导入/导出结构：

> 1. 编译器分析 `import`/`export` 语句，构建依赖图

> 2. 标记未被引用的导出（dead code）

> 3. Rollup/Webpack 在打包时移除标记的代码

> 要求：必须是 ESM 格式（`import`/`export`），CommonJS 的 `require()` 是动态的无法静态分析

  

**Q42: Vite 的 `server.proxy` 配置原理？**

  

> 基于 `http-proxy` 中间件：

> 1. 拦截匹配 `/api` 前缀的请求

> 2. 将请求转发到 `target` 指定的地址

> 3. `changeOrigin: true` 修改请求头 `Host`

> 4. 解决开发环境跨域问题，生产环境需 Nginx 反向代理

  

**Q43: `@vitejs/plugin-vue` 如何处理 `.vue` 文件？**

  

> 1. 使用 `@vue/compiler-sfc` 解析 SFC

> 2. `<template>` → `@vue/compiler-dom` 编译为 render 函数

> 3. `<script>` → 处理 `ref`/`computed` 等语法糖，编译为 ES module

> 4. `<style>` → `scoped` 属性注入 `data-v-xxx` 选择器 → PostCSS 处理

> 5. 组装成一个 ES module 导出

  

**Q44: 如何配置 Vite 的代码分割？**

  

> 自动分割：路由懒加载（`import()`）自动生成独立 chunk。

> 手动分割（`vite.config.js`）：

> ```js

> build: {

>   rollupOptions: {

>     output: {

>       manualChunks: {

>         vendor: ['vue', 'vue-router'],

>         vant: ['vant'],

>       }

>     }

>   }

> }

> ```

  

**Q45: Vite 的 `define` 配置有什么用？**

  

> 在编译时替换全局常量，类似 Webpack 的 DefinePlugin：

> ```js

> define: { __VERSION__: JSON.stringify('1.0.0') }

> // 代码中 __VERSION__ 被替换为 "1.0.0"

> ```

  

**Q46: esbuild 和 Rollup 在 Vite 中的分工？**

  

> - **esbuild**（Go）：开发环境依赖预构建 + TS/JSX 编译，速度极快

> - **Rollup**（JS）：生产环境打包，插件生态更成熟，支持更灵活的代码分割

  

**Q47: 如何优化 Vite 生产构建体积？**

  

> 1. 路由懒加载 + `manualChunks` 合理拆分

> 2. 组件按需导入（本项目 `unplugin-vue-components`）

> 3. CSS 提取（Vite 默认启用）

> 4. `build.minify: 'terser'` 启用更激进的压缩（默认 esbuild 更快但压缩率稍低）

> 5. `build.target` 设为现代浏览器，减少 polyfill

  

**Q48: Vite 的 `glob import` 是什么？**

  

> `import.meta.glob` 批量导入文件：

> ```js

> const modules = import.meta.glob('./views/*.vue')

> // { './views/Home.vue': () => import('./views/Home.vue'), ... }

> ```

> 适用于批量注册路由、批量导入组件等场景。

  

**Q49: 什么是 `browserslist`？Vite 如何处理兼容性？**

  

> `browserslist` 定义目标浏览器范围。Vite 根据 `build.target` 和 `browserslist` 决定：

> - JS 语法转换程度

> - CSS 前缀添加

> - Polyfill 范围

> 本项目未配置 `browserslist`，使用默认值。

  

**Q50: `vite preview` 的作用？**

  

> 本地预览生产构建产物，用于在开发阶段验证打包结果是否正确：

> ```bash

> npm run build && npm run preview

> ```

  

### HTTP & 网络（51-65）

  

**Q51: Axios 拦截器的原理是什么？执行顺序是怎样的？**

  

> Axios 内部维护一个拦截器队列：

> ```

> [reqInterceptor2, reqInterceptor1, dispatchRequest, resInterceptor1, resInterceptor2]

> ```

> 通过 Promise 链串联，请求拦截器后进先执行（栈），响应拦截器先进先执行。

  

**Q52: Axios 的 CancelToken 和 AbortController 有什么区别？**

  

> - `CancelToken`：Axios 旧版取消方案（已废弃），通过 `source.token`

> - `AbortController`：Web 标准 API，`controller.abort()` + `signal`

> - 本项目 SSE 请求中两者结合：内部 `canceled` 标志 + `AbortController`

  

**Q53: SSE (Server-Sent Events) 和 WebSocket 的区别？**

  

| 特性 | SSE | WebSocket |

|------|-----|-----------|

| 方向 | 单向（服务器→客户端） | 双向 |

| 协议 | HTTP | 独立协议 `ws://` |

| 断线重连 | 浏览器内置自动重连 | 需手动实现 |

| 消息格式 | 文本（SSE 协议格式） | 文本/二进制 |

| 复杂度 | 简单 | 较复杂 |

| 适用场景 | 实时推送、AI 流式输出 | 聊天室、协同编辑、游戏 |

  

> 本项目用 SSE 正确：AI 流式回复只需服务器→客户端单向推送，SSE 更轻量且自动重连。

  

**Q54: SSE 协议的格式是怎样的？**

  

> 见 6.2.1 节。核心规则：

> - 字段（`event:`/`data:`/`id:`/`retry:`） + 空行 `\n\n` 分隔事件

> - `data:` 可以多行，最终合并

> - `event:` 不指定时默认为 `message` 事件

  

**Q55: `fetch` 读取流式响应的原理？**

  

> 1. `fetch()` 返回 `Response` 对象

> 2. `response.body` 是 `ReadableStream`

> 3. `getReader()` 获取 reader

> 4. 循环 `reader.read()` 获取 `{ done, value }`（`value` 是 `Uint8Array`）

> 5. `TextDecoder` 将字节解码为字符串

> 6. 本项目通过 `{ stream: true }` 参数增量解码

  

**Q56: 什么是跨域？有哪些解决方案？**

  

> 浏览器的同源策略限制。解决方案：

> 1. **CORS**：服务器设置 `Access-Control-Allow-Origin` 头

> 2. **代理**：开发环境 Vite proxy，生产环境 Nginx 反向代理（本项目方案）

> 3. **JSONP**：利用 `<script>` 不受同源限制（仅 GET）

> 4. **WebSocket**：不受同源策略限制

> 5. **postMessage**：iframe 间跨域通信

  

**Q57: HTTP 状态码 301/302/304/400/401/403/404/500 的含义？**

  

> - `301`：永久重定向

> - `302`：临时重定向

> - `304`：未修改（缓存有效）

> - `400`：请求参数错误（本项目 `/plan` 参数不全返回 400）

> - `401`：未认证（需登录）

> - `403`：无权限

> - `404`：资源不存在

> - `500`：服务器错误（本项目 AI 调用失败返回 500）

  

**Q58: GET 和 POST 的区别？**

  

> - GET：参数在 URL 中，有长度限制，可缓存，幂等

> - POST：参数在 Body 中，无长度限制，不可缓存，非幂等

> - 本项目的设计选择：`/plan` 和 `/chat` 都用 POST，因为参数包含中文且可能较长

  

**Q59: Cookie、SessionStorage、LocalStorage 的区别？**

  

| 特性 | Cookie | SessionStorage | LocalStorage |

|------|--------|---------------|-------------|

| 容量 | 4KB | 5MB | 5MB |

| 有效期 | 可设置 | 标签页关闭 | 永久 |

| 发送到服务器 | 自动携带 | 不发送 | 不发送 |

| 跨域 | 可配置 | 同源 | 同源 |

  

**Q60: HTTP 缓存的机制？强缓存和协商缓存的区别？**

  

> - 强缓存：`Cache-Control: max-age=3600`、`Expires`，不请求服务器直接使用缓存

> - 协商缓存：`ETag/If-None-Match`、`Last-Modified/If-Modified-Since`，发请求验证是否过期，304 表示使用缓存

  

**Q61: RESTful API 设计规范？本项目符合吗？**

  

> RESTful 核心：资源导向、HTTP 动词语义化。

> 本项目的 `/api/plan` 和 `/api/chat` 不是严格 RESTful（是 RPC 风格 `POST + 动作名`）。

> 更 RESTful 的设计：`POST /api/trips`（创建规划）、`GET /api/trips/:id`（查询规划）。

  

**Q62: API 版本管理策略？**

  

> 常见方式：

> 1. URL 版本：`/api/v1/plan`、`/api/v2/plan`

> 2. Header 版本：`Accept: application/vnd.api.v1+json`

> 3. Query 版本：`/api/plan?version=1`

> 本项目未实现版本管理。

  

**Q63: JWT Token 的认证流程？**

  

> 1. 客户端发送登录请求

> 2. 服务器验证后生成 JWT（含用户信息 + 过期时间 + 签名）

> 3. 客户端存储 JWT（通常 localStorage）

> 4. 后续请求在 `Authorization: Bearer <token>` 头中携带

> 5. 服务器验证签名和过期时间

> 本项目无认证体系，但请求拦截器中预留了 Token 注入点：

> ```js

> request.interceptors.request.use((config) => {

>   // 可在此添加：config.headers.Authorization = `Bearer ${token}`

>   return config

> })

> ```

  

**Q64: XSS 和 CSRF 攻击是什么？如何防范？**

  

> - **XSS**（跨站脚本）：注入恶意脚本 → 防范：输入转义、CSP、HttpOnly Cookie

> - **CSRF**（跨站请求伪造）：利用已登录状态发起恶意请求 → 防范：CSRF Token、SameSite Cookie、验证 Referer

> - 本项目前端使用 Vue 模板语法，自动转义 HTML，天然防御 XSS

  

**Q65: HTTPS 的握手过程？**

  

> 1. 客户端发送 ClientHello（支持的加密套件 + 随机数）

> 2. 服务器返回 ServerHello + 证书 + 公钥

> 3. 客户端验证证书 → 生成 pre-master key → 用公钥加密发送

> 4. 双方用三个随机数生成对称密钥

> 5. 后续通信用对称密钥加密

**Q67: `<style scoped>` 的原理和穿透方案？**

  

> 原理：编译时给每个选择器添加 `[data-v-xxx]` 属性选择器。

> 穿透：使用 `:deep()` 伪类修改子组件样式：

> // 编译为：[data-v-xxx] .van-button { ... }

**Q68: Flexbox 和 Grid 的区别？适用场景？**

  

> - Flexbox：一维布局（行或列），适合组件内部排列

> - Grid：二维布局（行+列），适合页面整体布局

> - 本项目大量使用 Flexbox：`display: flex` 用于消息布局、输入栏、卡片排列、板块头部等

  

**Q69: BFC（块级格式化上下文）是什么？如何触发？**

  

> BFC 是独立的渲染区域，内部元素不影响外部。

> 触发方式：`overflow: hidden`、`display: flow-root`、`position: absolute`、`float` 等。

> 应用场景：清除浮动、防止 margin 重叠、自适应两栏布局。

  

**Q70: CSS 动画 vs JS 动画的优劣？**

  

> - CSS 动画：GPU 加速（`transform`/`opacity` 不触发重排）、声明式、性能好；复杂序列控制弱

> - JS 动画：完全控制帧率/暂停/反向、可编程；在主线程运行，性能开销大

> - 本项目打字机光标用 CSS `@keyframes`，打字机效果用 JS `setInterval`（需要控制队列和停止）

  

**Q71: `transform` 和 `position` 做动画哪个性能好？为什么？**

  

> `transform` 好。因为 `transform` 只触发 **Composite（合成）** 阶段，不触发 Layout（重排）和 Paint（重绘）。

> 浏览器渲染流水线：Layout → Paint → Composite。

> 本项目的点击反馈 `transform: scale(0.97)` 使用 transform 而非修改 width/height。

  

**Q72: 移动端 1px 边框问题的原因和解决方案？**

  

> 原因：Retina 屏幕的 `devicePixelRatio` 为 2 或 3，1px CSS 显示为 2-3 物理像素。

> 方案：

> 1. `border: 0.5px`（部分浏览器支持）

> 2. `transform: scale(0.5)` + 伪元素

> 3. `box-shadow` 模拟

> 4. Vant 组件内部已处理此问题

  

**Q73: CSS 的 `env(safe-area-inset-bottom)` 是什么？**

  

> 用于适配 iPhone X+ 的刘海屏和底部 Home Indicator 安全区。

> 本项目聊天输入栏使用 `calc(50px + 12px + env(safe-area-inset-bottom, 0px))` 将输入栏上移，避免被 Home Indicator 遮挡。

  

**Q74: `z-index` 的层叠上下文规则？**

  

> 层叠上下文的创建：`position: relative/absolute` + `z-index` 不为 auto、`opacity < 1`、`transform`、`flex` 子项等。

> 同一层叠上下文内比较 `z-index`，不同层叠上下文比较父级。

> 本项目输入栏 `z-index: 10` 确保浮在消息列表之上。

  

**Q75: PostCSS 是什么？本项目使用了哪些 PostCSS 功能？**

  

> PostCSS 是 CSS 后处理器，通过插件转换 CSS。

> Vite 内置 PostCSS，但本项目未配置额外插件。

> 常见用途：Autoprefixer（自动加浏览器前缀）、postcss-nesting（嵌套语法）、postcss-px-to-viewport（移动端适配）。

**Q76: Express 中间件机制的原理？**

  

> 中间件是一个函数数组，每个中间件接收 `(req, res, next)`，调用 `next()` 进入下一个。

> 本质是一个责任链模式，请求依次经过各中间件处理。

> 本项目使用的中间件链：`express.json()` → `travelRouter`。

  

**Q77: Express 5 相比 Express 4 的变化？**

  

> - 原生支持 `async/await` 错误处理（不需要 `express-async-errors`）

> - 路由路径匹配更严格

> - 移除了一些废弃的 API

> - 更好的安全默认值

> 本项目使用 Express 5。

  

**Q78: 什么是单例模式？本项目如何应用？**

  

> ```js

> // travelService.js

> class TravelService { /* ... */ }

> module.exports = new TravelService()  // 导出实例，不是类

> ```

> 单例确保 LLM 连接只初始化一次，所有请求复用同一实例。

  

**Q79: LangChain 的 `ChatOpenAI` 如何适配 DeepSeek？**

  

> 通过 `configuration.baseURL` 指向 DeepSeek 的 API 端点，因为 DeepSeek 兼容 OpenAI SDK 的请求/响应格式。这是典型的适配器模式。

  

**Q80: LLM 的 `temperature` 参数是什么？本项目为什么设为 0.7？**

  

> `temperature` 控制输出随机性（0-2）：

> - 0：确定性输出，每次相同

> - 0.7：平衡创意和可靠性（本项目选择）

> - 1.0+：高随机性，适合创意写作

> - 旅行规划需要一定创意（推荐景点）但不能太随机（预算需准确），0.7 是合理的折中。

  

**Q81: `invoke` 和 `stream` 的区别？**

  

> - `invoke`：等待完整响应后返回（`await`）

> - `stream`：返回异步迭代器，通过 `for await (const chunk of stream)` 逐块获取

> - 本项目 `/plan` 用 `invoke`（需要完整结果做正则解析）

> - `/chat` 用 `stream`（逐块推送到 SSE，实现打字机效果）

  

**Q82: SSE 后端实现的关键点？**

  

> 1. 设置响应头：`Content-Type: text/event-stream` + `Cache-Control: no-cache` + `Connection: keep-alive`

> 3. 监听 `res.on('close')` 处理客户端断开

> 4. 错误处理：headers 未发送时返回 JSON，已发送时写 SSE error 事件

  

**Q83: 如何用正则从 Markdown 中提取结构化数据？**

**Q84: Prompt Engineering 在本项目中如何体现？**

**Q95: 如何将打字机效果抽取为可复用的 Composable？**

  

> ```js

> // composables/useTypewriter.js

> export function useTypewriter(speed = 20) {

>   let queue = []

>   let timer = null

>

>   function start(targetRef) {

>     stop()

>     timer = setInterval(() => {

>       if (queue.length === 0) return

>       targetRef.value += queue.shift()

>     }, speed)

>   }

>

>   function feed(text) {

>     for (const ch of text) queue.push(ch)

>   }

>

>   async function stop(targetRef) {

>     if (timer) { clearInterval(timer); timer = null }

>     if (queue.length > 0) {

>       targetRef.value += queue.join('')

>       queue = []

>     }

>   }

>

>   return { start, feed, stop }

> }

> ```
